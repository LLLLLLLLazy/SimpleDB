##  2. 事务、锁与并发控制

在开始之前，你应确保自己理解什么是事务，以及严格两阶段锁（strict two-phase locking）如何保证事务的隔离性与原子性。

本节后续部分会简要回顾这些概念，并说明它们与 SimpleDB 的关系。

###  2.1. 事务

事务是一组数据库操作（例如插入、删除和读取），它们以*原子方式*执行；也就是说，要么全部完成，要么全部不生效，并且对于数据库的外部观察者而言，这些操作看起来像是一个不可分割的单一动作。

###  2.2. ACID 属性

为了帮助你理解 SimpleDB 中的事务管理方式，我们简要回顾它是如何满足 ACID 属性的：

* **Atomicity（原子性）**：通过严格两阶段锁和谨慎的 buffer 管理来保证。
* **Consistency（一致性）**：数据库通过原子性保持事务一致性。其他一致性问题（例如键约束）不在 SimpleDB 的处理范围内。
* **Isolation（隔离性）**：通过严格两阶段锁保证。
* **Durability（持久性）**：通过 FORCE buffer 管理策略保证（见下文第 2.3 节）。


###  2.3. 恢复与 Buffer 管理

为了简化实现，我们建议你采用 NO STEAL / FORCE 的 buffer 管理策略。

课堂上已经讨论过，这意味着：

*  如果某个 dirty（已更新）页面被某个尚未提交的事务加锁，就不应将它从 buffer pool 中淘汰（这就是 NO STEAL）。
*  当事务提交时，应当强制把该事务弄脏的页面写回磁盘（这就是 FORCE）。
   
为了进一步简化，你可以假设 SimpleDB 在处理 `transactionComplete` 命令的过程中不会崩溃。注意，这三点意味着本实验中你**不需要**实现基于日志的恢复：因为你永远不需要 undo 已被淘汰的 dirty 页面（你不会淘汰 dirty page），也永远不需要 redo 已提交但未落盘的工作（你在提交时强制刷盘，并且提交过程中不会崩溃）。

###  2.4. 授予锁

你需要在 SimpleDB 中加入调用（例如在 `BufferPool` 中），使调用者能够代表某个事务，为某个特定对象请求或释放一把锁（共享锁或排他锁）。

我们建议以*页面*为粒度加锁；为了便于测试，请不要实现表级锁（虽然理论上也能做）。本文档其余部分以及我们的单元测试都假设你使用页面级锁。

你需要创建一些数据结构，用于跟踪每个事务持有哪些锁，并在事务请求锁时判断是否应当授予。

你需要实现共享锁和排他锁；其规则如下：

*  在事务读取对象之前，必须先拥有该对象的共享锁。
*  在事务写入对象之前，必须先拥有该对象的排他锁。
*  多个事务可以同时持有某对象的共享锁。
*  某对象的排他锁在任意时刻只能被一个事务持有。
*  如果事务 *t* 是对象 *o* 上唯一持有共享锁的事务，那么 *t* 可以将其在 *o* 上的共享锁升级为排他锁。

如果事务请求的锁无法立刻授予，你的代码应当*阻塞*，等待该锁变得可用（即被另一个线程中的事务释放）。
你要特别注意锁实现中的竞争条件，认真思考并发调用会如何影响行为。
（你可能需要阅读 Java 中关于 <a href="http://docs.oracle.com/javase/tutorial/essential/concurrency/sync.html">
同步</a> 的内容。）

###  2.5. 锁的生命周期

你需要实现严格两阶段锁（strict 2PL）。这意味着：事务在访问任何对象之前都应先获取合适类型的锁，并且在事务提交之前不应释放任何锁。

幸运的是，SimpleDB 的设计允许你在 `BufferPool.getPage()` 中，在读取或修改页面前获取锁。因此，与其在每个算子中额外加入锁调用，我们建议直接在 `getPage()` 中获取锁。根据你的实现方式，也许根本不需要在别处再显式加锁；这点需要你自己确认。

在读取任何页面（或元组）之前，你需要先获取该页面（或元组）的*共享锁*；在写入之前，则需要先获取*排他锁*。你会注意到，`Permissions` 对象已经在 `BufferPool` 中四处传递；这些对象表示调用方希望在被访问对象上获取哪种锁（我们已经为你提供了 `Permissions` 类的代码）。

请注意，`HeapFile.insertTuple()` 和 `HeapFile.deleteTuple()` 的实现，以及 `HeapFile.iterator()` 返回的迭代器实现，都应该通过 `BufferPool.getPage()` 来访问页面。请仔细检查这些不同调用场景传入的权限对象是否正确（例如 `Permissions.READ_WRITE` 或 `Permissions.READ_ONLY`）。你也可能需要再检查一下 `BufferPool.insertTuple()` 和 `BufferPool.deleteTupe()` 的实现，确认它们对访问过的页面调用了 `markDirty()`（你应在 Lab 2 中已经这么做了，只是我们当时没有专门测这一点）。

在获取锁之后，你还需要考虑何时释放锁。很明显，为了保证严格 2PL，应当在事务提交或中止之后释放该事务关联的全部锁。
不过，也可能存在某些场景，在事务结束前提前释放锁是合理的。例如，在扫描页面寻找空 slot 的场景中，你可以在确认某页没有空位之后立即释放该页上的共享锁（见下文）。

###  2.6. 实现 NO STEAL

事务所做的修改只有在提交后才会写入磁盘。这意味着，中止一个事务时，我们可以通过丢弃 dirty page 并从磁盘重新读取它们来回滚。因此，我们必须**不能淘汰 dirty page**。这种策略称为 NO STEAL。

你需要修改 `BufferPool` 中的 <tt>evictPage</tt> 方法。具体来说，它绝不能淘汰 dirty page。如果你的淘汰策略本来想淘汰的是一个 dirty page，那么你必须找到其他可淘汰页面作为替代。如果 buffer pool 中所有页面都是 dirty 的，则应抛出 <tt>DbException</tt>。如果你淘汰的是一个 clean page，请注意某些事务可能仍持有该页面的锁，你需要在实现中妥善处理。

###  2.7. 事务完成

在 SimpleDB 中，每次查询开始时都会创建一个 `TransactionId` 对象。这个对象会被传递给参与该查询的各个算子。当查询结束时，会调用 `BufferPool` 的 `transactionComplete` 方法。

调用该方法会根据参数 `commit` 的值，选择*提交*或*中止*事务。在执行过程中，任意算子都可能抛出 `TransactionAbortedException`，表示发生了内部错误或死锁。我们提供的测试代码会创建相应的 `TransactionId`，正确传递给你的算子，并在查询结束时调用 `transactionComplete`。`TransactionId` 类本身我们已经实现好了。

###  2.8. 死锁与中止

在 SimpleDB 中，事务可能发生死锁（如果你不理解原因，建议阅读 Ramakrishnan & Gehrke 中有关死锁的章节）。你需要检测这种情况，并抛出 `TransactionAbortedException`。

检测死锁的方法有很多。最简单的做法之一是超时策略：如果一个事务在给定时间内没有完成，就将其终止。更严谨的做法是实现课堂上讲过的等待图（wait-for graph）上的环检测。在这种方案中，你可以周期性检查依赖图，或在每次尝试授予新锁时检查图中是否存在环；一旦有环，就必须中止某些事务。检测到死锁后，还需要决定如何打破它。假设事务 *t* 正在等待某把锁时，你检测到了死锁。如果你足够“凶狠”，可以把 *t* 正在等待的所有事务全部中止；这样可能会撤销大量工作，但可以保证 *t* 继续前进。另一种做法是直接中止 *t*，给其他事务让路。这样做的代价是上层用户代码需要重新执行事务 *t*。

另一类方法是利用事务的全局顺序来避免显式构建等待图。出于性能考虑，这类方法有时更受欢迎，但它可能会误杀原本能够成功完成的事务。典型例子包括 WAIT-DIE 和 WOUND-WAIT。
